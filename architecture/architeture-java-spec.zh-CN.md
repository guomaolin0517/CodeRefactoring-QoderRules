## 三、分层架构规范

### 3.1 分层职责

> **【强制】** 架构分层：前端展现层 → 控制层（Controller）→ 业务逻辑层（Service/BO）→ 数据访问层（DAO）→ 数据库层，**禁止**跨层调用（如 Controller 直接调用 DAO）。

| 分层 | 职责 | 依赖 |
|------|------|------|
| 控制层（Controller） | 仅负责请求接收、参数校验、结果封装，不包含业务逻辑 | Service 层 |
| 业务逻辑层（Service/BO） | `Service` 为接口，`ServiceImpl` 为实现类；`BO` 封装核心业务逻辑，`Service` 调用 `BO` 完成业务流程 | DAO 层 |
| 数据访问层（DAO） | 仅负责数据库操作，不包含业务逻辑，依赖 `DO` 对象，SQL 写在 Mapper 文件中 | DO 对象 |

> **【强制】** 对象传输规则：`DTO`（跨层传输）、`BO`（业务逻辑封装）、`DO`（数据库映射），**禁止** `DO` 直接对外暴露。

### 3.2 分层编码要求

> **【强制】** 各分层组件需继承框架统一基类（如 `CtjController`、`CtjService`），保障统一特性（如异常处理、日志记录）。

> **【强制】** `DTO`/`DO`/`BO` 字段与业务含义、数据库字段一致，**禁止**冗余无关字段，字段类型匹配（如数据库 `datetime` 对应 Java `LocalDateTime`）。
