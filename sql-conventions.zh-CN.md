## 二、命名规范

# [规则 1] 表名命名

> **【强制】** 所有命名使用**小写字母**，单词间**下划线 `_`** 分隔，**禁止**大写、特殊字符（`@`、`#`）、中文或拼音。

> **【禁止】** 表名使用数据库关键字（如 `order`、`user`、`table`），如需使用需加反引号（`` `order` ``）。

# [规则 2] 字段名命名

> **【强制】** 字段名：小写字母 + 下划线分隔，见名知意，**禁止**模糊命名（`col1`、`data`）。

> **【强制】** 主键字段：统一命名 `id`或`表名简称_id`（单主键），类型为 `number(19)`，使用雪花算法十进制字符串，**禁止**复合主键，**禁止**自增ID。

> **【强制】** 通用字段（所有表必含）：

| 字段名 | 类型 | 约束 / 说明 |
|--------|------|-------------|
| `rid` | `varchar2(40)` | 数据唯一记录号，全库唯一，记录创建时赋值，以后保持不变|
| `create_time` | `varchar2(17)` | 非空，`yyyyMMddHHmmss` |
| `update_time` | `varchar2(17)` | 非空，`yyyyMMddHHmmss` |


> **【禁止】** 使用外键。

> **【强制】** 枚举类字段（如状态）：使用 `number(3)` 类型，通过注释说明枚举值含义（**禁止**使用 `varchar2`）。

| 示例 | 说明 |
|------|------|
| `order_status` | `0` = 待支付，`1` = 已支付，`2` = 已取消 |

## 三、字段设计规范

### [规则 3] 字段类型选择

> **【强制】** 优先使用占用空间小的类型，避免资源浪费：

| 数据类型 | 推荐类型 | 说明 |
|----------|----------|------|
| 整型 | `number` | 根据数值范围选择，如用户年龄用 `tinyint`，订单 ID 用 `bigint` |
| 金额 | `number(12,2)` | 精确到分，**禁止**使用 `BINARY_FLOAT`/`BINARY_DOUBLE`（避免精度丢失） |
| 字符串 | `char`/`varchar2` | 固定长度用 `char`，可变长度用 `varchar2`，指定合理长度 |
| 时间 | `varchar2(17)` | 统一使用，**禁止**用 `datetime`/`timestamp` 存储时间 |

> **【强制】** 字段长度按需设置，避免过度冗余：

| 字段 | 类型 | 说明 |
|------|------|------|
| 用户名 | `varchar2(32)` | - |
| 手机号 | `char(11)` | - |
| 身份证号 | `char(18)` | - |

> **【强制】** 非空字段必须加 `NOT NULL` 约束，默认值合理（如 `is_delete` 默认 `0`，`status` 默认 `0`）。

> **【禁止】** 使用 `NULL` 存储空字符串，空值统一用 `''`（字符串）或 `0`（数值）。

## 四、索引设计规范

### [规则 4] 索引类型与创建原则

> **【强制】** 主键默认创建聚簇索引。

> **【强制】** 高频查询字段（如 `user_id`、`order_no`、`status`）创建普通索引。

> **【强制】** 联合索引遵循 **"最左前缀原则"**，将高频筛选字段放在左侧。

```sql
-- 示例
CREATE INDEX idx_user_status_create_time ON assp_order_main (user_id, status, create_time);
```

> **【强制】** 分页查询字段（`page_num`、`page_size`）需配合排序字段创建索引，避免分页结果不一致。

> **【禁止】** 过度索引（单表索引不超过 5 个），避免插入 / 更新时索引维护开销。

> **【禁止】** 在低基数字段（如 `is_delete`、`status` 仅含 2 个值）创建索引。

> **【禁止】** 索引字段使用函数（如 `TRUNC(create_time)`），避免索引失效。

## 五、SQL 编写规范

### [规则 5] 基础编写规则

> **【强制】** SQL 关键字**大写**（`SELECT`/`INSERT`/`UPDATE`/`DELETE`/`WHERE`/`JOIN`/`ON`/`ORDER BY`/`LIMIT`），表名 / 字段名**小写**，格式对齐，提高可读性。

> **【强制】** **禁止**使用 `SELECT *`，需明确查询字段（减少数据传输、避免字段变更影响）。

> **【强制】** `WHERE` 子句中**禁止**使用函数（如 `TRUNC(create_time) = '2024-05-01'`），避免索引失效。

> **【强制】** 批量操作（`INSERT`/`UPDATE`/`DELETE`）需控制批次大小（单次不超过 **1000 条**），避免数据库压力。

> **【强制】** 分页查询必须包含 `ORDER BY`（基于主键或索引字段），避免分页结果混乱。

> **【禁止】** 笛卡尔积查询（无 `ON` 条件的 `JOIN`），避免数据量过大导致性能崩溃。

> **【禁止】** SQL 注入风险：使用参数绑定（`#{param}`），**禁止**字符串拼接 SQL（`${param}` 仅用于表名 / 字段名动态场景，需严格校验）。

### [规则 6] 性能优化规则

> **【强制】** 避免长事务，减少锁竞争。

> **【强制】** `UPDATE`/`DELETE` 语句必须包含 `WHERE` 条件（除非明确全表更新 / 删除，需审批）。

> **【建议】** 高频查询的小表（如字典表）可适度冗余字段，减少 `JOIN` 操作。

> **【禁止】** 使用TRUNCATE，避免数据丢失。

## 六、安全规范 
### [规则 7] SQL安全性规则 [DISABLED]

> **【强制】** 敏感字段（密码、身份证号、银行卡号）需加密存储：
- 密码：**BCrypt** 加密
- 身份证号：**AES** 加密

> **【强制】** 查询敏感数据时需关联数据权限，确保用户只能查询所属范围的数据。

> **【强制】** **禁止**直接执行外部传入的 SQL 语句，所有动态 SQL 需通过框架提供的安全方式实现（如 MyBatis 动态 SQL）。

> **【强制】** 定期清理临时表数据，避免垃圾数据堆积。
